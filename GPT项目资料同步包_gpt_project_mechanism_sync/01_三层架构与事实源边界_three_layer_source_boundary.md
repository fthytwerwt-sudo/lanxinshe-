# 三层架构与事实源边界

## 文件定位

本文件规定账号记忆、GPT Project、GitHub `main`、Codex 四个层级之间的职责边界。

本文件只承载事实源边界机制，不承载当前项目事实。

## 三层 + 执行层定义

### 1. 账号记忆

账号记忆是跨项目长期协作偏好，用于保存用户稳定的工作方式和偏好。

它可以影响沟通方式和默认判断习惯，但不能覆盖当前项目事实。

### 2. GPT Project

GPT Project 是当前项目的配合机制层。

它应该承载：

- 协作协议。
- 判断顺序。
- 下发格式。
- 质量保障机制。
- 澄清闸门。
- 六层需求确认机制。
- Codex 执行模板。

它不应该承载：

- 项目状态。
- 执行报告。
- 阶段结果。
- API 状态。
- 素材路径。
- 验收结论。
- 客户需求细节。
- 仓库 `AGENTS.md` 原文。
- 执行日志原文。

### 3. GitHub main

GitHub `main` 是当前项目事实源。

项目事实包括：

- 项目状态。
- 执行报告。
- 阶段结果。
- 验收标准。
- 客户需求。
- API 状态。
- 代码。
- 测试结果。
- 风险记录。
- 决策记录。

### 4. Codex

Codex 是执行落库层。

Codex 负责在明确边界内读取文件、检查现状、执行任务、生成报告、写入仓库、验证、commit（提交）、push（推送），并把结果证据化回报。

## AGENTS.md 的位置

`AGENTS.md` 是仓库接手入口，不是 GPT Project 上传材料。

当当前仓库存在 `AGENTS.md` 时：

1. GPT / Codex 接手仓库任务时应先读取它。
2. `AGENTS.md` 中的当前项目事实仍属于 GitHub `main` 事实层。
3. `AGENTS.md` 中可复用的协作习惯可以抽象为机制后进入 GPT Project。
4. 不允许把旧项目 `AGENTS.md` 原文或旧项目事实复制到当前项目机制包。

## 冲突处理

- 用户本轮输入优先级最高。
- 机制冲突时，按 GPT Project 当前机制包执行。
- 项目事实冲突时，按 GitHub `main` 当前文件和验证证据执行。
- 历史记忆、旧项目经验、外部资料只能作为参考层，不能覆盖当前事实。

## 停止线

- 无法区分机制和事实时，不写入 GPT Project。
- 无法验证当前事实时，不写 `已确认`。
- 旧项目事实不得进入当前机制包。
- 当前项目事实不得进入 GPT Project 机制包。

## 一句话执行口径

账号记忆管长期偏好，GPT Project 管配合机制，GitHub `main` 管当前事实，Codex 管执行落库。
