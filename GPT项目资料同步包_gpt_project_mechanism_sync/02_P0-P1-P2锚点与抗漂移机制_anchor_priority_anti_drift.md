# P0-P1-P2 锚点与抗漂移机制

## 文件定位

本文件规定当前项目判断优先级，防止任务被历史记忆、旧项目经验、外部资料或聊天漂移带偏。

本文件只承载优先级机制，不承载项目事实。

## 默认优先级

1. `P0`：用户本轮明确目标、边界、禁止项、完成标准。
2. `P1`：当前仓库 GitHub `main` 文件、验证结果、commit（提交）/ push（推送）/ remote HEAD（远端最新提交指针）证据。
3. `P2`：历史聊天、账号记忆、旧项目机制、外部资料、研究摘要。

冲突时按 `P0 > P1 > P2` 执行。

## 默认动作

- GPT 下发 Codex 前，先写清 P0、P1、P2。
- Codex 执行前，先确认 workspace（当前工作目录 / 工作区）、branch（分支）、allowed files（允许文件）、forbidden files（禁止文件）、dirty status（未提交状态）。
- 任何未验证内容必须标 `待验证`。
- 任何基于经验的判断必须标 `推测` 或 `通用建议`。
- `P2` 只能提示方向，不能冒充当前项目事实。

## 旧项目迁移边界

旧项目可以迁移：

- 协作机制。
- 判断顺序。
- 下发格式。
- 质量保障机制。
- 澄清闸门。
- 六层需求确认机制。
- Codex 执行模板。

旧项目不能迁移：

- 项目身份。
- 项目状态。
- 素材路径。
- 模型选择。
- API 状态。
- 验收结果。
- 业务方向。
- 完成状态。
- 执行日志。
- 产物能力结论。

## 停止线

- P0 明确禁止的事情，不因 P1/P2 存在而执行。
- P1 没有证据的能力，不因 P2 经验写 `已确认`。
- P2 和 P1 冲突时，不融合、不折中，先回到 P0/P1。

## 一句话执行口径

先锁本轮输入，再守当前仓库事实，最后参考历史和外部资料；旧项目只给机制，不给事实。
