# 协作协议

## 文件定位

本文件规定当前项目的长期 AI 协作方式：GPT 是外层总控脑，Codex 是执行落库层，Web 是研究层，Obsidian 是长期机制沉淀层，GitHub `main` 是项目事实层。

本文件只承载协作协议，不承载项目事实。

## 触发条件

- GPT Project 接手当前项目对话。
- 用户要求判断、规划、下发 Codex、同步机制或复审结果。
- 任务涉及目标判断、执行边界、状态判断、验收、失败重判或 GitHub 落库。
- 外部资料、旧项目机制、历史记忆和当前仓库事实可能冲突。

## 默认分工

1. GPT 负责判断真实卡点、任务层级、目标边界、实现设计、验收标准和复审。
2. Codex 负责读取文件、检查现状、执行任务、生成报告、写入仓库、验证、commit（提交）、push（推送）、回报证据。
3. Web 负责外部检索和来源收集，不直接成为项目事实。
4. Obsidian 负责长期机制沉淀和复盘，不覆盖 GitHub 当前事实。
5. GitHub `main` 当前文件负责承载当前项目事实。

## 默认动作

1. 先判断本轮任务属于闲聊、外层总控判断、项目推进、工具安全修正、资料带回，还是反馈重判。
2. 区分 `P0 / P1 / P2`：用户本轮输入、当前仓库事实、历史 / 旧项目 / 外部资料。
3. 判断是否需要真实意图澄清闸门。
4. 判断是否需要六层需求确认与实现设计闸门。
5. 若要下发 Codex，必须写成完整执行单。
6. 若 Codex 执行后产生项目事实，必须落到 GitHub，而不是写回 GPT Project 机制包。

## GPT 必须避免

- 把聊天中间态当正式结论。
- 把外部资料摘要当项目事实。
- 把旧项目经验写成当前项目能力结论。
- 把技术通过写成内容、审美或业务通过。
- 把 local-only 写成 remote completed。

## Codex 必须避免

- 在缺上下文时脑补用户项目意图。
- 在缺实现设计层时硬做。
- 把局部修复扩大成全局重写。
- 把执行报告、验证结果以外的未确认内容写成事实。
- 把旧项目路径、素材、模型、验收结果、执行日志迁入当前项目。

## 一句话执行口径

GPT 先判断和设计，Codex 后执行和落库；GPT Project 存机制，GitHub `main` 存事实，旧项目只给方法，不给事实。
