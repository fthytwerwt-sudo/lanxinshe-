# 路线重判与失败后改线机制

## 文件定位

本文件规定当前项目执行失败、验证失败、用户反馈不对或路线冲突时，如何分层归因并改线。

本文件只承载失败处理机制，不承载具体失败记录。

## 触发条件

- 同一路线连续失败。
- 验证脚本、构建、push（推送）或 remote readback（远端回读）失败。
- 用户反馈“不对 / 空 / 差点意思 / 不顺 / 怪怪的”。
- 技术通过但内容、审美、人感或业务目标未通过。
- 继续执行会扩大边界或混入项目事实。

## 失败层级

| 失败层 | 处理方式 |
|---|---|
| 目标层 | 回到真实意图澄清 |
| 机制层 | 重写规则或触发条件 |
| 实现设计层 | 改 primary_route / fallback / probe |
| 流程层 | 调整执行步骤 |
| 执行层 | 修文件、脚本、命令并重跑验证 |
| 权限层 | blocked，不硬做 |
| 验收层 | 补标准或进入用户回审 |

## 默认动作

1. 先记录失败现象和失败证据。
2. 判断失败层级。
3. 列出保留、放弃、降级、新 route、fallback。
4. 判断是否需要重新过六层需求确认。
5. 如果是权限、账号、API、余额或 secret 问题，直接 blocked。
6. 如果只是技术通过，不得写成内容或业务通过。

## 停止线

- push 失败不写 completed。
- 权限/API/余额/账号失败不硬做。
- 路线错不继续调参数。
- 用户反馈方向错，不只修执行细节。
- 能力未 probe，不写 `已确认`。

## 一句话执行口径

失败不是继续重试的理由；先分层，再决定修复、降级、改线或 blocked。
