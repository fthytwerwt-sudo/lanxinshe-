# GPT 与 Codex 自动补全及质量保障机制

## 文件定位

本文件规定 GPT 与 Codex 如何把方向、机制、项目事实和执行任务补到可执行、可验证、可回滚、可同步。

本文件只承载质量保障机制，不承载项目事实。

## 触发条件

- 用户给出方向型输入。
- GPT 准备下发 Codex。
- Codex 读取后发现目标、边界、入口、验证、完成标准不完整。
- 机制包、仓库事实、历史记忆之间存在冲突。
- 用户要求“别只总结，要能执行”。

## GPT 补全责任

GPT 必须补清：

- 用户真实目标。
- 成功标准。
- 失败标准。
- 停止条件。
- P0 / P1 / P2。
- 六层需求确认。
- Implementation design（实现设计）。
- 验收标准。
- 风险和 blocked 条件。

## Codex 补全责任

Codex 可以在当前边界内补：

- 文件 inventory（文件清单）。
- 影响面检查。
- 必须读取文件。
- 输入输出字段。
- 执行步骤。
- 验证命令。
- 报告格式。
- 缺口清单。

Codex 不得补：

- 用户真实业务意图。
- 核心路线选择。
- 审美判断。
- 当前项目未验证能力结论。
- 旧项目事实。

## 质量保障清单

- 有目标和边界。
- 有真实意图澄清。
- 有六层需求确认。
- 有实现设计层。
- 有 primary_route。
- 有 fallback_route。
- 有 capability_status。
- 有 probe / validation 决策。
- 有 `blocked_need_implementation_design_layer`。
- 有输入字段、输出字段、字段来源。
- 有执行入口和验证命令。
- 有完成标准和失败标准。
- 有 commit / push / remote readback 边界。
- 已排除项目事实进入 GPT Project 机制包。

## 停止线

- 缺真实意图，不下发 Codex。
- 缺实现设计层，不下发 Codex。
- 缺验证方式，不写 completed。
- 缺 GitHub 当前事实证据，不写 `已确认`。
- 只允许机制同步时，不写入项目事实。

## 一句话执行口径

GPT 把方向和实现设计补清，Codex 把任务补成可执行、可验证、可落库；缺实现设计、缺验证或缺远端闭环都不是完成。
